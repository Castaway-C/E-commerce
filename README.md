# It's Mygo

一次买够（It's Mygo）学生实训项目骨架。

项目结构以 `backend`、`user-frontend`、`admin-frontend`、`docs`、`scripts` 为主，后续按《实现设计说明书.md》逐步补充业务。

## 技术栈

- 后端：FastAPI + SQLAlchemy async + Alembic + Redis + Celery
- 用户端：React + TypeScript + Vite + Ant Design
- 管理端：React + TypeScript + Vite + Ant Design/ProComponents
- Python：3.12.10

## 主要目录

- `backend/`：后端 FastAPI 工程
- `user-frontend/`：用户端响应式 Web 工程
- `admin-frontend/`：管理端 Web 工程
- `docs/`：项目文档与接口约定
- `scripts/`：开发辅助脚本
- `tests/`：跨端或验收测试说明，后端单元测试在 `backend/tests/`

## 三人开发分工建议

建议三人固定主责，但不要完全割裂。每个阶段都先对齐接口文档，再分别开发，最后一起联调。

| 成员 | 主责 | 兼顾内容 |
|---|---|---|
| 成员 A | 后端基础设施、数据库模型、业务接口 | OpenAPI、接口文档、后端测试 |
| 成员 B | 用户端前端、商品浏览、购物车、订单流程 | 用户体验、接口联调、移动端适配 |
| 成员 C | 管理端前端、后台运营页面、数据看板 | 权限联调、测试用例、项目文档维护 |

协作时每个功能建议按“接口契约 -> 后端接口 -> 前端页面 -> 联调验收”的顺序推进。接口字段一旦多人引用，变更前要先同步文档并通知其他成员。

## 开发顺序规划

### 第 0 阶段：项目骨架与公共约定

目标是让三个人能在统一目录、统一命名、统一接口规则下开工。

- 确认 `docs/api/` 中的接口路径、响应结构、错误码和分页规则。
- 确认用户端与管理端账号体系分离，普通用户不进入后台。
- 后端补齐统一响应、异常处理、日志、鉴权依赖、数据库会话、Redis 客户端。
- 前端补齐请求封装、路由守卫、基础布局、环境变量读取。
- 三人共同维护 `AGENTS.md` 和 `docs/conventions.md`，避免后续风格分裂。

### 第 1 阶段：账号与权限基础

这是所有业务的前置能力。

- 后端：普通用户注册登录、管理端登录、JWT 刷新、密码加密、基础权限依赖。
- 用户端：登录、注册、个人资料入口。
- 管理端：登录页、当前管理员信息、菜单和权限点占位。
- 联调重点：普通用户 token 和管理员 token 不混用；接口无权限时返回统一错误。

### 第 2 阶段：商品、店铺与上传

先让平台有可浏览、可管理的商品基础数据。

- 后端：商家、分类、商品、SKU、图片上传、商品状态流转。
- 用户端：首页、商品列表、商品详情、店铺页。
- 管理端：商品列表、商品编辑、分类管理、商品审核占位。
- 联调重点：商品价格统一使用分；SKU 库存和商品状态以后端为准。

### 第 3 阶段：购物车、结算与订单主链路

优先打通 P0 主交易链路。

- 后端：购物车、结算预校验、创建订单、模拟支付、订单列表和详情。
- 用户端：购物车、结算页、订单列表、订单详情、支付结果页。
- 管理端：订单列表、订单详情、商家发货。
- 联调重点：`client_order_token` 幂等、库存不足、订单状态不允许操作等错误处理。

### 第 4 阶段：收货、评价与售后

让交易闭环完整。

- 后端：确认收货、评价发布、售后申请、退款状态流转。
- 用户端：确认收货、发表评价、申请售后。
- 管理端：评价审核、售后审核、退货收货确认。
- 联调重点：评价必须来自已完成订单；售后状态和订单主状态不要混乱。

### 第 5 阶段：社区与种草闭环

实现项目特色功能，但要在订单主链路稳定后再做。

- 后端：帖子、评论、点赞、话题、种草帖商品关联、内容审核状态。
- 用户端：社区广场、帖子详情、发帖、评论点赞。
- 管理端：帖子审核、评论审核、内容下架。
- 联调重点：种草帖关联商品需要校验购买记录；内容先审后发。

### 第 6 阶段：促销与积分

在基础交易稳定后接入价格计算增强能力。

- 后端：优惠券、满减、限时特价、积分账户、积分流水。
- 用户端：领券、我的优惠券、结算页优惠选择、积分展示。
- 管理端：优惠券模板、满减活动、限时活动配置。
- 联调重点：价格计算顺序统一为“限时特价/拼团价 -> 满减 -> 优惠券 -> 积分抵扣”。

### 第 7 阶段：异步任务、WebSocket 与数据看板

补齐实训加分项和运营体验。

- 后端：Celery 超时取消订单、自动确认收货、优惠券过期、报表导出。
- WebSocket：客服消息、订单通知、限时活动对时。
- 管理端：经营看板、销售趋势、商品排行、订单状态分布。
- 联调重点：异步任务要幂等；WebSocket 不承载下单扣库存等关键交易逻辑。

### 第 8 阶段：测试、演示数据与答辩准备

最后统一收口，保证演示可控。

- 准备初始化脚本和演示账号。
- 为注册登录、商品浏览、购物车、下单、支付、发货、收货、评价补测试用例。
- 整理接口文档、项目结构说明、核心流程图和分工说明。
- 三人分别准备自己负责模块的讲解材料，但演示流程统一串联。

## 分支与提交建议

- 默认保护主分支，三人各自从 `main` 拉分支开发。
- 分支命名建议：`feature/backend-auth`、`feature/user-cart`、`feature/admin-products`。
- 小步提交，提交信息说明做了什么，例如 `add auth api skeleton`。
- 合并前至少让另一名成员看一遍接口影响和页面影响。
- 每次合并前确认：能启动、无明显语法错误、相关文档已更新。
