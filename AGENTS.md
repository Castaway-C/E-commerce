# AGENTS.md

这是一个学生实训项目，不完全追求生产级，但要结构清晰、可维护、方便三人协作开发。

## 项目定位

- 项目名：一次买够（It's Mygo）。
- 类型：社交新零售电商平台。
- Python 环境：3.12.10。
- 当前目标：先把商城基础闭环、管理端运营基础、社区与促销基础能力做稳，再逐步补完整页面和细节流程。
- 不要为了“看起来生产级”而提前引入复杂架构、复杂依赖或过度抽象。

## 文档优先级

- `需求规格说明书.md` 不要随意改动，优先遵循。
- `实现设计说明书.md` 是编码依据；如与需求或现有实现冲突，先指出问题，再决定是否修改。
- 编码时必须同步相关接口文档，尤其是 `docs/api/*.md`。
- 如果接口字段、状态、错误码、权限边界发生变化，文档必须同一轮更新。
- AGENTS.md 作为当前开发进度和协作约定文档，需要随阶段进展实时更新。

- 社区发帖、评论、点赞、管理端隐藏管理能力；帖子和评论发布后直接展示，种草来源下单和确认收货后积分奖励基础闭环已实现。
- 基础积分流水已实现，种草奖励通过统一积分服务写入 `points_log`。
- 管理端用户列表、订单列表/详情、看板汇总。
- 管理端订单基础 CSV 导出已实现；平台运营导出全平台订单，商家运营只能导出本店订单。
- 管理端商品权限边界、商品编辑、SKU 价格/库存编辑、上下架。
- 后台操作日志基础能力已实现，平台运营可查询；已接入商品监管、订单发货、售后处理、批量发券等关键动作。
- 商家入驻流程已修正为商家自助注册：`merchant_pending` 登录后仅能查看/重新提交自己的入驻资料，平台审核通过后创建店铺并升级为 `merchant_operator`；被拒绝后不限制重新提交次数，重新提交后回到 `pending`；平台运营可查询后台账号、启停账号、重置密码，脚本 `backend/scripts/create_admin.py` 仍用于本地初始化平台账号。

- 平台不手动创建店铺或商品。店铺来自商家入驻审核通过，商品由 `merchant_operator` 创建；平台只负责分类创建、商家入驻审核和全平台商品监管。

当前前端已重写为 React + TypeScript + Ant Design 5 的 PC 端商城/后台页面。用户端 `/` 是有分类、商品卡、左图右文详情、购物车、订单、社区的商城页；管理端拆成 `/platform` 平台管理、`/merchant` 商家管理、`/merchant-apply` 商家入驻，平台账号和商家账号使用两套 localStorage 会话，可同时登录。正常使用时商品、订单、售后、优惠券、社区、账号等信息必须显示在对应业务区；所有需要操作的 ID（商品 ID、SKU ID、分类 ID、店铺 ID、订单 ID、售后 ID 等）必须直接显示，接口返回 JSON 只用于报错排查，且保留最近多条记录。

## 权限和账号边界

- 普通用户使用 `/api/v1/auth/*` 和用户端接口。
- 后台管理员使用 `/api/v1/admin/*`。
- `platform_operator` 是平台运营，可查看和管理全平台数据。
- `merchant_pending` 是商家自助注册后、平台审核前的待审核账号，只能登录、查看自己的入驻申请状态、重新提交入驻资料，不能操作商品、订单、促销。
- 商家入驻被拒绝后不限制重新提交次数；调用 `PUT /api/v1/admin/merchant/application/me` 会把申请状态重新置为 `pending` 并清空原拒绝原因；已审核通过的申请不能再重新提交修改。
- `merchant_operator` 是商家运营，只能操作自己绑定 `merchant_id` 的数据。
- 店铺必须通过商家入驻审核创建，平台不能手动创建店铺。
- 商品必须由 `merchant_operator` 创建，平台不能手动创建商品；平台拥有分类创建、商品上下架和监管权限。
- 新增管理端接口时必须考虑商家管理员边界，不能默认全平台可见。
- 订单、商品、促销、社区内容管理等接口都要明确是用户端能力还是管理端能力。

## 接口约定

- 统一响应格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

- 分页接口统一返回：

```json
{
  "list": [],
  "page": 1,
  "page_size": 20,
  "total": 0
}
```

- 常用错误码：
  - `40001`：参数错误。
  - `40002`：未登录或 token 无效。
  - `40003`：无权限或越权访问。
  - `40004`：数据不存在。
  - `40005`：业务冲突或当前资源不可用。
  - `40008`：状态不允许执行当前操作。

## 前端页面节奏

- 不要每新增一个小接口就立刻写前端页面。
- 当前阶段前端页面用于联调和人工验收，样式可以很简洁，但内容和布局必须清晰可用。
- 当前前端不是按钮墙或接口调试页面，要像日常商城/后台网页，视觉简洁但必须美观、分区清楚、操作顺手，不能退回无样式的基础组件拼装页。
- 即使是联调页面，也要有基础设计系统：品牌导航、页面首屏、信息卡片、状态标签、价格/ID 标签、清晰表格、合理留白和移动端响应。
- 正常信息必须显示在表格、列表、卡片、状态区或表单旁边；不要要求使用者通过接口返回 JSON 才能知道商品、订单、售后、优惠券、审核状态或账号信息。
- 有 ID 的或需要通过 ID 操作的内容必须在业务区直接展示 ID。
- “接口返回排查”区域可以保留，但必须折叠或放在页面底部，仅用于出问题时查看最近一次请求结果。
- 用户端 `/` 是普通用户商城流程页；管理端 `/platform` 是平台运营页，`/merchant` 是商家运营页，`/merchant-apply` 是商家入驻页，三者不要混在一个页面里。
- 当一个模块形成可测试闭环后，再补一批最简页面。
- 页面中尚未实现的内容可以用占位展示，但不要误导为已完成能力。
- 后端接口稳定后，再逐步补完整交互、状态提示和页面细节。

## AI 编码注意事项

- 每次编码前先读相关现有代码和文档，不要凭空猜结构。
- 发现设计文档明显不一致或会影响实现时，先指出问题。
- 编码后同步文档和测试。
- 不要提前实现复杂业务流程；先保证当前阶段闭环正确。
- 回答完成情况时说明改了哪些文件、验证了什么、还有哪些后续风险。
