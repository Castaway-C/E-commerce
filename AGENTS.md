# AGENTS.md

这是一个学生实训项目，不完全追求生产级，但必须结构清晰、可维护、方便三人协作开发。

## 项目定位

- 项目名：一次买够（It's Mygo）。
- 类型：社交新零售电商平台。
- Python 环境：3.12.10。
- 当前状态：电商平台基本功能已经实现，可进入后续全量开发阶段。
- 后续目标：对照需求书和设计书，把每个模块完整实现到可演示、可联调、可维护，而不是继续停留在占位或临时测试能力。
- 不要为了“看起来生产级”而提前引入复杂架构、复杂依赖或过度抽象。

## 文档优先级

- `需求规格说明书.md` 是功能目标和验收依据。
- `实现设计说明书.md` 是编码设计依据。
- 用户明确允许时，可以修订 `需求规格说明书.md` 和 `实现设计说明书.md`；修订后必须在回复中完整说明改动点。
- 编码时必须同步相关接口文档，尤其是 `docs/api/*.md`。
- 如果接口字段、状态、错误码、权限边界发生变化，文档必须同一轮更新。
- `AGENTS.md` 是当前进度、协作规则和上下文恢复文档，需要随阶段进展更新。

## 当前功能基线

当前项目可视为基础电商平台已实现：

- 用户与管理端认证分离，普通用户和后台管理员不能混用。
- 普通用户支持注册、登录、刷新、登出、资料查询。
- 管理端支持平台账号和商家账号同时登录、刷新、登出，平台端和商家端 token 分开存储。
- 商家自助注册入驻：`merchant_pending` 登录后仅能查看/重新提交自己的入驻资料；平台审核通过后创建店铺并升级为 `merchant_operator`；拒绝后不限制重新提交次数。
- 店铺名称保持唯一。注册、重新提交、审核通过时都要检查店铺名冲突并返回业务错误。
- 平台不手动创建店铺或商品。店铺来自商家入驻审核通过，商品由 `merchant_operator` 创建；平台负责分类、商家入驻审核、商品监管和内容治理。
- 商品、店铺、分类、SKU、上传能力已实现；商品由商家创建后默认上架，平台保留上下架监管能力。
- 购物车、结算、下单、模拟支付、取消未支付订单、发货、确认收货已实现。
- 库存变动已抽取统一库存服务，管理端手动调整、下单扣减、取消/超时取消回补都会写 SKU 库存流水。
- 支付超时取消、自动确认收货已有 service 方法、Celery 任务入口和 Celery beat 调度配置。
- 收货地址、订单地址快照、评价、售后退款基础流转已实现。
- 售后申请包含退款金额、原因分类；退款完成会同步支付单 `refunded` / `partial_refunded` 状态。
- 售后库存回补采用实训规则：确认收到退货且全额退款后，回补订单内全部 SKU 库存并写 `refund_restore` 流水；部分退款或未确认退货直接退款不自动回补库存。
- 按实现设计书 6.2，本项目不接入真实物流轨迹查询；物流只做商家填写物流公司/单号、用户确认收货的状态推进。
- 优惠券支持领取与下单抵扣，适用范围包括 `all/platform/merchant/category/product/sku`，已支持按范围金额计算门槛、模板编辑/停用、按用户 ID 批量发券、手动过期作废和 Celery 过期任务。
- 社区支持发帖、评论、点赞、平台隐藏管理；帖子和评论发布后默认公开。
- 种草帖校验已购商品，订单支持 `source_post_id`，确认收货后给种草帖作者积分奖励。
- 基础积分流水已实现，积分变动应通过 `points_service.change_points`。
- 管理端已有用户列表、后台账号管理、订单列表/详情、看板汇总、订单 CSV 导出、操作日志、促销管理、社区内容治理、评价内容治理和售后处理。
- 前端已是 React + TypeScript + Ant Design 5 的 PC 页面。用户端 `/` 是可用商城页；管理端拆成 `/platform` 平台运营、`/merchant` 商家运营、`/merchant-apply` 商家入驻。

## 重要业务规则

- 只有商家入驻需要事前审核。
- 商品、帖子、评论、评价均采用发布后管理；平台保留隐藏、下架、禁用、监管和操作日志能力。
- 普通用户使用 `/api/v1/auth/*` 和用户端接口。
- 后台管理员使用 `/api/v1/admin/*`。
- `platform_operator` 可管理全平台数据。
- `merchant_pending` 只能登录、查看自己的入驻申请状态、重新提交入驻资料，不能操作商品、订单、促销。
- `merchant_operator` 只能操作自己绑定 `merchant_id` 的数据。
- 新增管理端接口时必须考虑商家管理员边界，不能默认全平台可见。
- 订单、商品、促销、社区内容治理等接口都要明确是用户端能力还是管理端能力。
- 不实现真实物流轨迹查询；不要在后续任务中规划轨迹接口或轨迹页面。

## 后续开发总原则

- 后端扩展新功能时，前端必须同轮添加调用入口或页面入口。
- 每个模块后续都按完整需求开发，不能只做占位、半流程或只能靠接口返回查看结果。
- 一个功能完成的最低标准：后端模型/schema/service/router、前端页面或调用入口、接口文档、关键测试或验收说明同时完成。
- 价格计算、优惠券抵扣、积分变动、订单状态流转、库存流水、权限判断等可复用逻辑应集中在 service 或 helper 中，router 和页面不直接堆业务判断。
- 页面正常信息必须显示在表格、列表、卡片、状态区或表单旁边；接口返回 JSON 只用于排查。
- 有 ID 的或需要通过 ID 操作的内容必须在业务区直接展示 ID。
- 不要随意引入不必要的新依赖。

## 后续优先任务

后续任务以 `docs/development-roadmap.md` 为准。当前重点方向：

1. 账号与用户中心：个人信息页、用户资料编辑、详细地址、登录态刷新和切换账号后的数据清理。
2. 分类与商品：分类树完整展示、父子关系、排序说明、品牌/标签、规格管理、商品详情和评价展示增强。
3. 订单与售后：售后明细级退货数量、订单状态筛选、运营报表、更多异常状态测试。
4. 促销：满减、限时价、拼团、积分抵扣，并统一价格计算顺序。
5. 会员与积分：会员等级、积分账户、签到、积分流水前端页。
6. 社区与种草：种草绑定商品改为选择器、话题标签、用户主页、商家广告帖权限边界。
7. 管理端：权限点细化、操作日志扩展、数据看板、导出能力和平台/商家筛选。
8. 技术能力：Redis 库存预扣、WebSocket 客服/通知、Celery 任务可观测性。

## 前端约定

- 用户端 `/` 是普通用户商城流程页。
- 管理端 `/platform` 是平台运营页。
- 管理端 `/merchant` 是商家运营页。
- 管理端 `/merchant-apply` 是商家入驻页。
- 三者不要混在一个页面里。
- 当前前端不是最终正式 UI，但必须像日常商城/后台网页，保持基础美观、分区清楚、操作顺手。
- 后续新增后端功能时，至少要提供可发现的菜单、按钮、表单或列表入口。
- 页面中尚未完整实现的内容可以标注为待完善，但不能误导为已完成能力。
- 当前已知前端待增强：
  - 地址字段需要更详细，例如省市区、街道、邮编、标签。
  - 用户个人信息页需要独立完善。
  - 种草绑定商品后续应改为勾选商品，不继续依赖逗号分隔 ID。
  - 分类需要完整树形展示、父级关系和排序含义说明。
  - 切换分类、切换账号、刷新 token 后的数据刷新需要继续增强。

## 接口约定

统一响应格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

分页接口统一返回：

```json
{
  "list": [],
  "page": 1,
  "page_size": 20,
  "total": 0
}
```

常用错误码：

- `40001`：参数错误。
- `40002`：未登录或 token 无效。
- `40003`：无权限或越权访问。
- `40004`：数据不存在。
- `40005`：业务冲突或当前资源不可用。
- `40008`：状态不允许执行当前操作。

## 测试和验证

最近稳定验证结果：

- 后端测试：`26 passed, 2 warnings`。
- 后端编译：`python -m compileall backend\app backend\tests` 通过。
- 用户端 TypeScript：`.\node_modules\.bin\tsc.cmd -b` 通过。
- 管理端 TypeScript：`.\node_modules\.bin\tsc.cmd -b` 通过。
- 两个 warning 是 FastAPI `on_event` 既有弃用提示。
- 当前 Codex 沙箱内完整 `npm run build` 会在 Vite/esbuild 读取配置阶段失败：`Cannot read directory "../../..": Access is denied`；这属于沙箱权限限制，用户本机可自行复测完整构建。

后端改动后优先运行：

```powershell
.\.venv\Scripts\python.exe -m pytest backend/tests -q
.\.venv\Scripts\python.exe -m compileall backend\app backend\tests
```

注意不要直接对整个 `backend` 目录执行 `compileall`，避免遍历本地虚拟环境。

前端改动后运行：

```powershell
cd user-frontend
npm run build

cd admin-frontend
npm run build
```

如果 Codex 沙箱因 Vite/esbuild 权限问题阻止完整构建，可先运行 `.\node_modules\.bin\tsc.cmd -b` 做 TypeScript 检查，并在最终回复说明构建限制。

## 数据库和迁移注意

- 当前开发环境主要依赖 ORM `create_all` 建表，适合新增表。
- 给已有表新增字段时要谨慎，因为 `create_all` 不会自动改旧表结构。
- 如果必须改已有表字段，要同时说明本地开发库如何处理，必要时提供迁移或清库建议。
- 当前 SQLite 开发库已有轻量补列逻辑；继续给旧表加字段时要同步 `backend/app/db/session.py`。
