# 后续全量开发计划

当前项目已经具备电商平台基本功能闭环。后续开发不再以“先做最小闭环”为目标，而是按模块完整实现需求，并保持后端、前端、文档、测试同步推进。

## 开发交付标准

每个后续功能至少同时完成以下内容：

- 后端：模型、schema、service、router、权限判断、错误码和关键测试。
- 前端：用户端或管理端可发现的菜单、页面、表单、列表、状态提示和接口调用。
- 文档：同步更新 `docs/api/*.md`、必要时更新 `实现设计说明书.md`、`需求规格说明书.md`、`docs/dev-setup.md`。
- 验收：给出可手工走通的流程，重要状态和异常有测试或说明。

除非项目负责人明确要求，否则不要只新增后端接口而不接前端，也不要只做页面静态占位。

## 三人协作方式

建议按模块纵向切分，而不是长期按“只写后端/只写前端”切分。

- 成员 A 主看后端模型、service、测试和接口文档。
- 成员 B 主看用户端页面、用户流程和用户体验。
- 成员 C 主看管理端页面、运营流程、权限联调和项目文档。

每个功能开始前先开一个小范围设计说明：本次涉及哪些接口、哪些页面、哪些状态、哪些权限、哪些文档。完成后一起按验收流程走一遍。

## 近期必须补齐

### 1. 账号、会话与用户中心

- 用户个人信息页：头像、昵称、手机号、邮箱、性别、生日等资料展示和编辑。
- 用户登出、平台登出、商家登出已具备入口，后续需要完善 token 过期后的自动刷新和失败回退。
- 切换用户后清理旧用户购物车、订单、优惠券、地址、社区选择状态。
- 地址扩展为省、市、区、街道、详细地址、邮编、标签、默认地址。
- 后端补充资料字段校验、唯一性规则和安全更新逻辑。

验收标准：切换账号后页面不残留上一个用户数据；用户资料和地址能正常增删改查；接口文档字段完整。

### 2. 分类、商品与店铺

- 分类树完整展示父子关系、排序含义和启停状态；用户端展示后端最新分类，不只展示预设分类。
- 商品规格管理增强，支持更清晰的规格名、规格值和 SKU 组合。
- 商品详情补齐店铺信息、评价列表、库存状态、图片展示和关联推荐。
- 店铺页补齐店铺信息、店铺商品、店铺优惠券和运营状态。
- 商品筛选、搜索、排序、价格区间、销量/上新等常见商城能力。
- 品牌、营销标签、推荐位可按需求逐步加入。

验收标准：平台创建的新分类能在用户端和商家端同步可见；商家能完整维护商品信息；用户能按常见条件找到商品。

### 3. 购物车、订单与支付

- 购物车数量修改、移除、批量选择和失效商品提示继续完善。
- 结算页价格明细要完整展示商品金额、优惠、运费、积分抵扣和实付金额。
- 订单状态机和异常路径补齐：取消、超时取消、支付失败、发货后售后、自动确认收货。
- 当前只做模拟支付；如后续接沙盒支付，需要补支付单状态、回调幂等和对账说明。
- Redis 库存预扣和防超卖可作为后续技术增强加入。

验收标准：同一商品加购、减购、移除、重新下单都稳定；订单状态变化前后端显示一致。

### 4. 售后与评价

- 售后明细级退货数量、退款金额计算和库存回补规则细化。
- 售后详情页补齐处理记录、原因、凭证、金额和状态时间线。
- 评价补齐追评、图片、评分维度和平台隐藏能力。
- 明确评价和售后与订单状态的关系，避免重复申请或越权处理。

验收标准：部分退款、全额退款、退货退款、仅退款等路径有清晰状态和页面反馈。

### 5. 促销与价格计算

- 当前优惠券已实现，后续补齐满减、限时价、拼团和积分抵扣。
- 统一价格计算顺序，建议为：活动价/拼团价 -> 满减 -> 优惠券 -> 积分抵扣。
- 所有价格计算逻辑必须集中在后端 service，前端只展示结果和明细。
- 管理端要提供活动配置、启停、适用范围、库存/名额、时间和状态管理。

验收标准：同一订单的优惠明细可解释，前后端金额一致，边界场景有测试。

### 6. 会员与积分

- 会员等级、成长值、积分账户、签到、积分抵扣。
- 积分变动统一复用 `points_service.change_points`，不要直接改 `user.points`。
- 用户端补积分账户页和积分流水；管理端补积分调整或查询权限。
- 种草奖励继续接入统一积分流水。

验收标准：积分来源、消耗、撤销都有流水；积分余额与流水汇总一致。

### 7. 社区与种草

- 种草绑定商品改为商品选择器，不再依赖手输逗号分隔 ID。
- 话题标签、帖子详情、用户主页、作者信息、关联商品卡片。
- 商家广告帖权限边界和管理端治理入口。
- 帖子、评论、评价继续保持发布后管理，平台可隐藏或处理违规内容。

验收标准：种草帖只能绑定已购买商品；来源下单和确认收货后奖励能被清晰追踪。

### 8. 管理端与经营报表

- 平台和商家管理端继续保持路由、会话和权限分离。
- 平台补全用户、商家、商品、订单、售后、促销、社区的筛选、详情和导出。
- 商家补全本店商品、订单、售后、优惠券、经营数据。
- 操作日志继续扩展更多关键动作，并支持按角色、商家、对象类型筛选。
- 数据看板补销售趋势、商品排行、订单状态分布、售后概览、优惠效果。

验收标准：平台能看全局，商家只能看本店；所有关键管理动作有日志。

### 9. WebSocket、任务与基础设施

- WebSocket 补客服聊天、订单通知、活动倒计时对时。
- Celery 任务补可观测日志、失败重试和幂等保护。
- Redis 用于 token 黑名单、库存预扣、热点缓存等能力时，要保持接口兼容。
- Alembic 迁移在模型稳定后逐步补齐，减少对 SQLite 补列逻辑的依赖。

验收标准：异步任务重复执行不破坏数据；WebSocket 不承载关键交易扣库存逻辑。

## 当前已知前端问题归档

- 分类切换和切回全部已做基础自动刷新，后续仍需完整分类树页面。
- 用户、平台、商家已补登出入口，后续需完善 token 过期刷新体验。
- 地址字段仍需更详细。
- 用户个人信息页仍需独立实现。
- 种草绑定商品后续应改为勾选商品。
- 部分路由目前仍指向综合页，后续应拆成正式页面。

## 文档维护要求

- `docs/api/*.md` 记录接口路径、方法、参数、响应、错误码、分页、权限。
- `docs/dev-setup.md` 记录启动、账号、清库、联调和人工验收流程。
- `docs/conventions.md` 记录命名、接口、权限、前后端同步规则。
- `AGENTS.md` 记录当前项目状态和 AI 协作恢复信息。
- 主文档发生需求或设计调整时，最终回复必须明确列出修订内容。
