# 一次买够（It's Mygo）实现设计说明书

> 本文基于《需求规格说明书》整理，用于正式编码前的实现设计确认，也可直接交给 AI 作为搭建项目框架和生成代码的依据。
> 《需求规格说明书》不做修改；本文在实现层面对需求中的角色、端侧、状态机、库存、支付、审核、权限和数据模型做落地约定。

---

## 一、项目定位与实现原则

### 1.1 平台命名

- 中文名：一次买够
- 英文名：It's Mygo
- 项目代码名：`its-mygo`

### 1.2 实现目标

平台是一个社交新零售电商系统，核心链路为：

```text
浏览商品/社区种草 -> 加购/立即购买 -> 结算支付 -> 商家发货 -> 用户收货评价 -> 社区转化与积分奖励
```

实现时优先保障以下目标：

| 优先级 | 目标 | 说明 |
|---|---|---|
| P0 | 主交易链路可跑通 | 注册登录、商品浏览、购物车、下单、支付回调/模拟支付、发货、收货、评价 |
| P0 | 后台可运营 | 商品、订单、用户、社区内容、促销活动、数据看板均可管理 |
| P1 | 社交种草闭环 | 已购用户发布种草帖，其他用户通过帖子购买后给原作者发放积分 |
| P1 | 促销闭环 | 优惠券、满减、限时特价可参与结算 |
| P1 | 库存一致性 | Redis 预扣 + MySQL 事务扣减 + 失败即时补偿 + 超时回滚 |
| P2 | 拼团、图床、复杂报表 | 可先实现基础版，再逐步增强 |

### 1.3 与需求文档的关键对齐

需求文档要求用户端“以 PC 端为主，适当兼容移动端”。因此实现设计中将用户端定义为响应式 Web，而不是只面向手机的 H5。

后台管理端服务对象是平台运营和商家运营；普通用户不进入后台。需求中的“普通用户”作为系统角色存在于用户端，不作为管理端登录角色处理。

## 二、系统总体架构

系统采用前后端分离架构，划分为用户端、管理端和后端服务三部分。

```text
用户端响应式 Web              管理端 Web
React + TypeScript            Ant Design Pro
        │                            │
        └──────────────┬─────────────┘
                       │
                    Nginx
          /api/* -> FastAPI HTTP API
          /ws/*  -> FastAPI WebSocket
          /static/uploads/* -> 静态资源
                       │
                 FastAPI 应用
    ┌──────────────────┼──────────────────┐
    │                  │                  │
 Routers           Services          WebSocket
 参数校验/分发      业务逻辑/事务       客服/通知/对时
    │                  │                  │
    └──────────────────┼──────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
      MySQL          Redis          Celery
   核心持久化       缓存/购物车       异步任务
   事务/索引        库存预扣/锁       超时取消/发券/导出
```

### 2.1 架构边界

| 层级 | 职责 |
|---|---|
| 用户端 | 商品、店铺、社区、购物车、下单、支付、个人中心 |
| 管理端 | 平台运营和商家运营的中后台管理 |
| FastAPI 路由层 | 接收请求、鉴权、参数校验、调用 Service，不写复杂业务 |
| Service 层 | 业务规则、事务边界、跨模块协作 |
| Model/Repository 层 | SQLAlchemy 模型、查询封装、Redis 访问 |
| Celery | 延迟任务、异步任务、报表导出、活动预热 |
| WebSocket | 客服聊天、订单通知、限时活动对时，不承担下单扣库存等关键交易逻辑 |

## 三、技术栈

### 3.1 用户端响应式 Web

| 分类 | 选型 | 说明 |
|---|---|---|
| 框架 | React + TypeScript | 与需求一致 |
| 构建 | Vite | 启动快，适合实训快速开发 |
| UI | Ant Design 5 + CSS 响应式布局 | PC 优先，同时兼容移动端 |
| 状态管理 | Zustand | 管理登录态、购物车、全局 UI |
| 路由 | React Router v6 | 页面路由 |

| 图表 | ECharts | 用户端少量可视化可复用 |
| 代码规范 | ESLint + Prettier | 前端统一格式 |

用户端不采用 Ant Design Mobile 作为主 UI 方案，避免与“PC 端为主”的需求冲突。移动端适配通过 CSS media query、弹性布局和少量移动样式完成。

### 3.2 管理端 Web

|---|---|---|

| 权限 | RBAC 简化实现 | 后端返回角色、菜单和权限点 |
| 状态管理 | Zustand 或 Ant Design Pro 内置 initialState | 二选一，推荐保持简单 |
| 图表 | ECharts | 数据看板 |

### 3.3 后端

| 分类 | 选型 | 说明 |
|---|---|---|
| Web 框架 | FastAPI | 自动 OpenAPI 文档，异步友好 |
| ORM | SQLAlchemy 2.0 async | 成熟稳定 |
| 数据迁移 | Alembic | 管理表结构变更 |
| 数据库 | MySQL 8.0 | 核心业务数据 |
| 缓存/锁/队列 | Redis 7.x | 购物车、缓存、库存预扣、Celery Broker |
| 异步任务 | Celery | 超时订单取消、自动收货、批量发券、报表导出 |
| 鉴权 | JWT + Bcrypt | access/refresh token |
| 文件处理 | local uploads + 可切换图床 | 先本地，联调时可使用图床 URL |

## 四、角色与权限设计

### 4.1 用户体系

| 角色 | 登录端 | 说明 |
|---|---|---|
| consumer | 用户端 | 普通消费者/会员 |
| merchant_operator | 管理端 | 商家运营，只能管理自己店铺的数据 |
| platform_operator | 管理端 | 平台运营，可管理全平台数据和审核 |

普通用户和后台管理员分表管理：

- `user`：消费者账号。
- `admin_user`：后台账号，包含 `role` 和可选 `merchant_id`。
- 商家账号必须绑定 `merchant_id`，所有商品、订单、帖子、客服会话都要按商家隔离。

### 4.2 权限边界

| 功能 | consumer | merchant_operator | platform_operator |
|---|---|---|---|
| 浏览商品/社区 | 是 | 可使用测试用户账号浏览 | 是 |
| 下单/评价/发帖 | 是 | 否 | 否 |
| 商品管理 | 否 | 仅本店商品 | 全平台 |
| 订单发货 | 否 | 仅本店订单 | 可查看全平台，可协助处理 |
| 内容审核 | 否 | 可查看本店广告帖状态 | 全平台审核 |
| 促销配置 | 否 | 可配置本店活动，平台可限制范围 | 全平台 |
| 数据看板 | 否 | 本店数据 | 全平台数据 |
| 管理员管理 | 否 | 否 | 是 |

## 五、模块划分与边界

### 5.1 商品中心 Products

负责商品、SKU、分类、品牌、店铺展示、商品评价。

| 内容 | 设计 |
|---|---|
| 核心实体 | `merchant`, `product`, `sku`, `category`, `brand`, `product_image`, `product_tag`, `review` |
| 依赖 | 基础模块，不依赖订单和促销 |
| 被依赖 | 订单、促销、社区、客服 |
| 关键规则 | 商品必须归属商家；SKU 管理价格和库存；评价必须来自已完成订单 |

商品状态：

```text
draft 草稿 -> pending_audit 待审核 -> on_sale 在售
on_sale -> off_sale 下架
on_sale -> sold_out 售罄
pending_audit -> audit_rejected 审核拒绝
```

### 5.2 交易与订单 Orders

负责购物车、结算、订单、支付、发货、收货、退款退货。

| 内容 | 设计 |
|---|---|
| 核心实体 | `cart_item`, `order`, `order_item`, `order_log`, `payment`, `refund`, `shipping_address` |
| 依赖 | 商品、促销、用户积分、社区来源追踪 |
| 关键规则 | 下单时以后端实时价格和库存为准；金额使用分为单位存储 |

订单主状态：

```text
pending_payment 待支付
wait_group 已支付待成团
pending_shipment 待发货
shipping 配送中
pending_receipt 待收货
completed 已完成
cancelled 已取消
after_sale 售后中
closed 已关闭
```

订单主状态转换：

```text
pending_payment --支付成功--> pending_shipment
pending_payment --拼团支付成功--> wait_group
pending_payment --用户取消/支付超时--> cancelled
wait_group --拼团成功--> pending_shipment
wait_group --拼团失败/超时--> closed

pending_shipment --商家发货--> shipping
shipping --系统流转--> pending_receipt
pending_receipt --用户确认/7天自动确认--> completed

pending_receipt --申请售后--> after_sale
completed --申请售后--> after_sale

after_sale --商家拒绝/用户取消售后--> 原状态(pending_receipt 或 completed)
after_sale --部分退款完成--> completed
after_sale --全额退款完成--> closed
```

支付状态单独记录：

```text
unpaid -> paid -> refunded/partial_refunded
unpaid -> closed
```

支付与订单关系：

- 普通单商家结算：1 个 `payment` 对应 1 个 `order`。
- 多商家购物车结算：1 个 `payment` 对应多个按商家拆分的 `order`。
- 拼团参团：每个参团用户生成自己的 `payment` 和 `order`，再挂到同一个 `group_buy_order`。
- 通过 `payment_order` 关联表记录 payment 与 order 的对应关系、支付分摊金额和退款金额。

退货退款状态单独记录，避免把订单主状态拉得过长：

```text
pending_approval 待商家审核
approved 商家同意
rejected 商家拒绝
returned 用户已寄回
received 商家已收货
refunded 已退款
closed 已关闭
```

售后记录必须保存 `origin_order_status`，用于拒绝售后或用户取消售后时恢复订单主状态。

### 5.3 促销引擎 Promotions

负责优惠券、满减满折、限时特价、拼团。

| 内容 | 设计 |
|---|---|
| 核心实体 | `coupon_template`, `user_coupon`, `full_discount_activity`, `flash_activity`, `flash_activity_sku`, `group_buy_activity`, `group_buy_order`, `group_buy_participant` |
| 依赖 | 商品 SKU、会员等级 |
| 被依赖 | 结算价格计算 |

促销优先级：

```text
限时特价/拼团价 -> 满减满折 -> 优惠券 -> 积分抵扣
```

规则说明：

- 限时特价使用普通 SKU 库存，但活动价需要单独记录在 `flash_activity_sku`。
- 若需要防超卖，仍使用 Redis SKU 库存预扣，不建立独立活动库存。
- 拼团商品可复用商品和 SKU 信息，但下单入口独立，不进入购物车。
- 拼团订单未成团前不发货，超时未成团自动退款。

优惠券校验规则：

| 规则 | 说明 |
|---|---|
| 适用范围 | `scope_type` 支持 all/platform、merchant、category、product、sku；对应范围 ID 存入关联表 |
| 门槛 | `min_amount_cent` 为最低消费门槛，按优惠券适用范围内商品金额计算 |
| 优惠类型 | `discount_type` 支持 amount 固定金额、percent 折扣 |
| 叠加 | 默认同一订单只能使用 1 张优惠券；是否可与满减叠加由 `can_stack_full_discount` 控制，默认允许 |
| 限时特价 | 默认可与优惠券叠加；若活动设置 `disable_coupon=true`，则不可用券 |
| 拼团价 | 默认不可再使用优惠券、满减和积分抵扣，避免规则复杂 |
| 会员限制 | 可配置 `min_user_level`，不满足等级不可领取或使用 |
| 有效期 | 支持固定起止时间；过期后 Celery 将用户券置为 expired |

`user_coupon` 状态机：

```text
unused 未使用 -> used 已使用
unused 未使用 -> expired 已过期
unused 未使用 -> void 作废
used 已使用 -> refunded_rollback 订单退款后返还/回滚
```

### 5.4 社交与内容 Community

负责社区帖子、评论、点赞、话题、种草来源追踪、客服消息。

| 内容 | 设计 |
|---|---|
| 核心实体 | `post`, `post_image`, `post_product`, `comment`, `like`, `topic_tag`, `post_topic_tag`, `cs_conversation`, `cs_message` |
| 依赖 | 用户、商品、订单 |
| 关键规则 | 帖子、评论、评价采用先审后发；种草帖必须校验已购 |

帖子类型：

```text
normal 普通交流贴
grass 种草帖，必须关联已购商品
merchant_ad 商家广告帖，必须由商家后台账号发布或平台代发
```

内容状态：

```text
pending_audit 待审核 -> published 已发布
pending_audit -> rejected 审核拒绝
published -> hidden 下架/隐藏
```

种草积分规则：

- 用户 A 发布已审核通过的种草帖并关联商品。
- 用户 B 从帖子点击商品进入详情或结算，订单记录 `source_post_id` 和 `source_user_id`。
- 用户 B 订单完成后，Celery 发放种草积分给用户 A。
- 同一订单只奖励一次；同一用户购买同一商品是否重复奖励由配置控制，默认只奖励首次有效转化。

### 5.5 会员与账户 Users

负责消费者账号、会员等级、积分、签到、地址。

| 内容 | 设计 |
|---|---|
| 核心实体 | `user`, `user_level`, `user_address`, `points_account`, `points_log`, `sign_in_record` |
| 依赖 | 订单完成金额、社区转化 |

会员等级：

```text
normal 注册即有
silver 累计消费满 500 元
gold 累计消费满 2000 元
diamond 累计消费满 5000 元
```

积分流水必须记录业务来源：

```text
sign_in 签到
purchase_reward 购物返利
grass_reward 种草奖励
order_deduction 下单抵扣
refund_rollback 退款回滚
manual_adjust 后台调整
```

### 5.6 数据与后台管理 Admin

负责后台菜单、管理员账号、数据看板、导出、操作日志。

| 内容 | 设计 |
|---|---|
| 核心实体 | `admin_user`, `admin_menu`, `operation_log` |
| 依赖 | 所有业务模块 |
| 关键规则 | 平台运营看全量，商家运营只看本店 |

## 六、核心业务流程

### 6.1 用户下单流程

```text
1. 前端提交选中商品/SKU/数量/地址/优惠券/积分使用量，允许包含多个商家的商品。
2. 后端读取 SKU 最新价格、库存、商品状态和活动状态。
3. Promotion Service 计算限时价、满减、优惠券、积分抵扣。
4. 按商家拆分订单草稿，生成同一个 payment 预支付单和多个 order。
5. Redis Lua 逐个预扣 SKU 库存，并记录本次已预扣的 sku_id/quantity 列表。
6. MySQL 事务创建 payment、order、order_item 记录，扣减 MySQL 库存，写订单日志，清理购物车已购买项。
7. 如果 MySQL 事务失败或抛出异常，立即执行 Redis 库存补偿，将第 5 步已预扣数量加回，并记录补偿日志。
8. 创建 Celery 延迟任务，15 分钟后检查待支付 payment。
9. 前端进入支付页，调用沙盒支付或模拟支付接口。
10. 支付成功后，同一 payment 下全部订单变为 pending_shipment，通知对应商家后台。
11. 若超时未支付，Celery 关闭 payment，取消其下所有订单并回滚 Redis/MySQL 库存。
```

实现约定：

- 下单必须幂等，前端传 `client_order_token`，后端防重复提交。
- 价格计算结果必须落库到订单明细，后续商品改价不影响历史订单。
- 订单金额字段统一使用整数分：`total_amount`, `discount_amount`, `pay_amount`。
- `POST /orders` 返回 `payment_id`、`payment_no` 和 `orders[]`，前端以 payment 为单位展示“去支付”。
- 若暂时不实现聚合支付，可在接口层限制一次结算只能包含一个商家；但当前推荐实现聚合支付，减少用户多次付款。

### 6.2 发货与收货流程

```text
pending_shipment -> 商家填写快递公司/快递单号 -> shipping
shipping -> 系统展示配送中，同时进入待收货页面 -> pending_receipt
pending_receipt -> 用户确认收货或发货 7 天后自动确认 -> completed
completed -> 用户可评价商品
```

实训项目不接入真实物流查询，快递状态由商家和用户手动推进即可。

### 6.3 退款退货流程

```text
1. 用户对 completed 或 pending_receipt 订单申请退货退款。
2. 系统生成 refund 记录，保存 origin_order_status，订单主状态变为 after_sale。
3. 商家审核：同意则 refund.approved，拒绝则 refund.rejected，订单恢复原状态。
4. 用户填写退货单号，refund.returned。
5. 商家确认收货，refund.received。
6. 系统执行退款事务：payment 状态变更、库存回滚、积分回滚、refund.refunded。
7. 订单关闭或恢复为 completed，按是否全额退款决定。
```

售后状态处理：

| 场景 | 订单主状态结果 |
|---|---|
| 商家拒绝售后 | 恢复 `origin_order_status` |
| 用户取消售后 | 恢复 `origin_order_status` |
| pending_receipt 订单全额退款 | `closed` |
| completed 订单全额退款 | `closed` |
| completed 订单部分退款 | `completed` |
| 多订单聚合支付中单个订单退款 | 只更新该订单和对应 `payment_order` 退款金额，payment 可变为 partial_refunded |

### 6.4 社区种草闭环

```text
1. 用户 A 购买并完成商品。
2. 用户 A 发布 grass 类型帖子，选择已购商品。
3. 后端校验已完成订单，创建帖子，状态为 pending_audit。
4. 平台运营审核通过，帖子公开。
5. 用户 B 从帖子进入商品并下单，订单记录 source_post_id。
6. 用户 B 订单完成，Celery 发放种草积分给用户 A。
7. WebSocket 或站内通知告知用户 A 积分到账。
```

### 6.5 限时特价流程

```text
1. 平台或商家创建限时活动，配置活动时间、活动 SKU、活动价。
2. 活动开始前 5 分钟，Celery Beat 预热活动 SKU 和库存缓存。
3. 前端进入活动页，通过 WebSocket 获取服务器时间并展示倒计时。
4. 用户下单时后端校验活动是否有效，并按活动价计算。
5. Redis Lua 预扣普通 SKU 库存，MySQL 事务最终扣减。
6. 活动结束后清理活动缓存。
```

### 6.6 拼团流程

```text
1. 用户选择拼团商品并直接发起拼团订单，不进入购物车。
2. 系统创建主订单 order，订单状态为 pending_payment，order_type=group_buy，并创建/关联 group_buy_order。
3. 用户支付成功后，主订单暂不进入 pending_shipment，订单主状态必须变为 wait_group，group_buy_order 状态为 grouping。
4. 其他用户通过拼团链接参团，各自生成独立 order 和 group_buy_participant。
5. 达到成团人数后，group_buy_order 变为 success，所有已支付参团订单进入 pending_shipment。
6. 超过成团截止时间未满员，Celery 将 group_buy_order 置为 failed；已支付的 wait_group 订单退款后进入 closed，未支付订单进入 cancelled，并回滚库存。
```

拼团实现约定：

- 主订单状态增加 `wait_group`，表示已支付但等待成团。
- `group_buy_order` 与 `order` 为 1:N：一个团包含多个用户订单。
- `group_buy_participant` 记录 `group_buy_order_id`、`order_id`、`user_id`、`paid_at`、`status`。
- 拼团价不参与普通购物车结算，不叠加优惠券、满减和积分抵扣。

| Token | 有效期 | 存储位置 | 用途 |
|---|---|---|---|
| access_token | 15 分钟 | 前端内存，必要时可用 sessionStorage | 请求鉴权 |
| refresh_token | 7 天 | HttpOnly + Secure + SameSite=Strict Cookie | 刷新 access_token |

刷新流程：

```text
接口返回 401 -> Axios 拦截器携带 Cookie 调用 /auth/refresh -> 获取新 access_token -> 重放原请求
refresh 失败 -> 清除登录态 -> 跳转登录
```

管理端和用户端登录接口分离，避免消费者 token 误用于后台：

- 用户端：`/api/v1/auth/login`

Cookie 约定：

- refresh_token 不向 JavaScript 暴露，降低 XSS 窃取长期凭证的风险。
- 登出时服务端清除 refresh_token Cookie，并将 access_token 的 jti 写入 Redis 黑名单。
- 如果本地开发未启用 HTTPS，可在开发环境临时关闭 Secure；联调和演示环境必须开启 Secure。

## 九、缓存、库存与异步任务

### 9.1 Redis 使用

| Key | 类型 | 用途 |
|---|---|---|
| `cart:{user_id}` | Hash | 购物车，field 为 sku_id，value 为数量 |
| `stock:sku:{sku_id}` | String | SKU 库存缓存 |
| `lock:order:{order_no}` | String NX EX | 下单幂等/短锁 |
| `coupon:{coupon_id}:claimed` | Set | 领券去重 |
| `flash:{activity_id}` | Hash/ZSet | 限时活动缓存 |
| `jwt:blacklist:{jti}` | String | 登出 token 黑名单，TTL 等于 token 剩余有效期 |

### 9.2 库存扣减策略

```text
读取 Redis 库存 -> Lua 原子预扣 -> MySQL 事务扣减 -> 成功提交
                                      └─ 失败/异常 -> 立即补偿 Redis
```

Lua 示例：

```lua
local stock = redis.call('GET', KEYS[1])
if not stock or tonumber(stock) < tonumber(ARGV[1]) then
  return 0
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1
```

MySQL 扣减建议使用乐观锁：

```sql
UPDATE sku
SET stock = stock - :quantity, version = version + 1
WHERE id = :sku_id AND stock >= :quantity AND version = :version;
```

补偿规则：

- Redis 预扣成功后，必须在内存中记录 `reserved_items = [{sku_id, quantity}]`。
- MySQL 事务提交失败、乐观锁扣减失败、订单创建异常或进程捕获业务异常时，立即遍历 `reserved_items` 执行 `INCRBY stock:sku:{sku_id} quantity`。
- 补偿成功与失败都要写日志；补偿失败时写入 `stock_compensation_log`，由定时任务重试。
- Celery 超时取消只处理“订单已创建但未支付”的场景，不替代 MySQL 事务失败时的即时补偿。
- 增加 `reconcile_stock_cache` 定时对账任务，每天或每小时扫描热点 SKU，对比 Redis 库存与 MySQL `sku.stock`，发现差异后按订单预占记录修正。

### 9.3 Celery 任务

| 任务 | 触发 | 延迟 | 说明 |
|---|---|---|---|
| `check_payment_timeout` | 创建订单 | 15 分钟 | 未支付则取消订单、回滚库存 |
| `auto_confirm_receipt` | 商家发货 | 7 天 | 自动确认收货 |
| `reward_grass_points` | 订单完成 | 立即 | 发放种草积分 |
| `expire_coupons` | 定时 | 每天 | 过期优惠券作废 |
| `batch_send_coupons` | 运营触发 | 立即 | 批量发券 |
| `warm_flash_activity` | 定时 | 活动前 5 分钟 | 预热活动缓存 |
| `clean_flash_activity` | 定时 | 活动结束后 | 清理活动缓存 |
| `check_group_buy_timeout` | 拼团创建 | 到期时 | 未成团退款 |
| `reconcile_stock_cache` | 定时 | 每小时/每天 | Redis 与 MySQL 库存对账，修复异常预占 |
| `export_data` | 后台请求 | 立即 | 生成 Excel |

所有任务必须具备幂等性，例如通过业务唯一键或状态判断避免重复处理。

## 十三、非功能设计

### 13.1 安全

| 项 | 方案 |
|---|---|
| 密码 | Bcrypt 加盐哈希 |
| 鉴权 | JWT HS256，区分用户端和管理端 |
| SQL 注入 | SQLAlchemy 参数绑定 |
| XSS | 后端过滤富文本危险标签，前端渲染转义 |
| CSRF | 业务 API 使用 Bearer access_token；refresh 接口使用 SameSite=Strict Cookie，必要时增加 CSRF token |
| 限流 | Redis 滑动窗口，登录接口 5 次/分钟，普通接口 60 次/分钟 |
| 文件上传 | MIME 校验、大小限制、随机文件名 |

### 13.2 日志与审计

- 所有请求生成 `request_id`。
- 管理端关键操作写入 `operation_log`。
- 订单状态变化写入 `order_log`。
- 异步任务记录任务 ID、重试次数、错误原因。

### 13.3 测试建议

P0 测试范围：

- 注册、登录、刷新 token。
- 商品列表、商品详情、SKU 选择。
- 加购、结算、下单、支付、超时取消。
- 商家发货、用户确认收货、评价。
- 发布种草帖、审核、通过帖子下单、积分奖励。
- 优惠券、满减、限时价参与价格计算。
- 商家账号只能访问本店数据。

## 十四、编码顺序建议

推荐按以下顺序让 AI 或小组成员实现：

1. 后端基础设施：配置、数据库、Redis、JWT、统一响应、异常处理。
2. 用户和后台账号：注册登录、角色权限、菜单。
3. 商家、分类、商品、SKU、上传。
4. 用户端商品浏览、店铺页、购物车。
5. 订单、支付模拟、库存预扣、Celery 超时取消。
6. 管理端商品、订单、发货。
7. 评价、社区帖子、评论、审核。
8. 种草来源追踪和积分奖励。
9. 优惠券、满减、限时特价。
10. 拼团、数据看板、Excel 导出。

## 十五、设计决策记录

| 决策项 | 最终选择 | 原因 |
|---|---|---|
| 用户端形态 | PC 优先响应式 Web | 与需求规格一致 |
| 用户端 UI | Ant Design 5 | PC 体验更完整，并与后台风格统一 |
| 后台角色 | platform_operator / merchant_operator | 普通用户不进入后台 |
| 商家模型 | 独立 `merchant` 表 | 支持店铺页、商家隔离、订单归属 |
| 金额存储 | 整数分 | 避免浮点误差 |
| 库存 | Redis 预扣 + MySQL 事务扣减 | 兼顾性能与一致性 |
| 限时特价库存 | 复用普通 SKU 库存 | 符合需求“无需独立库存” |
| 订单和售后状态 | 主状态与退款状态分离 | 避免状态机过度复杂 |
| WebSocket | FastAPI 原生 WebSocket | 满足客服、通知、倒计时即可 |
| Celery Broker | Redis | 减少中间件数量，适合实训 |
