 # 社交新零售电商平台 —— 需求规格说明书
 
 ## 一. 项目概述与定位
 
 本项目构建一个面向全国市场的社交新零售电商平台，以"线上下单 + 快递配送"为核心履约链路，融合社交裂变、内容驱动与智能促销三大引擎，重塑人货场连接方式。平台突破传统电商的单向货架逻辑，创新性地整合了社区内容分享、用户种草分享与多层级促销体系，形成"社交即流量、内容即转化、会员即资产"的增长闭环。
 
 系统采用前后端分离架构，端侧划分清晰，旨在通过社交裂变降低获客成本，依托精细化会员体系与私域运营提升用户复购与粘性。技术方案兼顾高并发场景的实战挑战与工程可落地性，是展示全栈开发能力与系统设计思维的优质实训载体。
 
 ### 1.1 核心技术栈（实训基准）
 
 - **前端核心**：React + TypeScript。顾客端采用响应式网页（以 PC 端为主，适当兼容移动端）；商家与运营管理端合并为一套基于 Ant Design Pro 的中后台 Web 系统。
 - **后端核心**：FastAPI（Python）。利用异步高并发特性，同时享受自动生成 Swagger API 文档的开发效率优势。
 - **数据与中间件**：
   - MySQL：核心业务数据存储（商品、订单、用户等），基于查询优化实现常规商品搜索。
   - Redis：热点数据缓存、购物车、库存预扣、Session 管理。
   - Celery：异步任务处理（超时订单取消、批量发券、报表生成）。
   - WebSocket：实时通信（客服聊天、限时特价倒计时对时、新订单通知）。
 
 ## 二. 角色与端侧划分
 
 为避免重复建设，系统划分为两个主要的前端工程：
 
 | 端侧 | 目标用户 | 核心职责 |
 |------|---------|---------|
| 用户端（Web） | 普通消费者、会员 | 浏览商品与店铺、参与社区互动、发布种草帖、下单支付、个人中心管理。 |
| 管理端（Web） | 商家、平台运营管理员 | 采用 RBAC（基于角色的访问控制）。商家负责上架商品、发货、发布广告贴；运营负责商家入驻审核、内容治理、活动配置、数据看板分析。 |
 
 ## 三. 核心业务闭环流转（业务亮点）
 
 1. **社区种草闭环**：用户 A 购买商品后在交流社区发布种草图文帖，并在帖子中关联已购商品；用户 B 浏览帖子后点击链接购买该商品；订单完成后，系统自动发放种草积分奖励至用户 A 的账户。
 2. **库存锁定与释放闭环**：用户提交订单后，Redis 预扣库存并创建订单，进入待支付锁定状态；Celery 启动延迟任务，若 15 分钟未完成支付则自动取消订单并回滚库存。
 3. **限时特价防超卖闭环**：限时特价活动开启后，前端通过 WebSocket 获取精确服务器时间；用户发起购买请求，后端校验商品常规库存并生成订单，保障活动期间价格准确生效。
 4. **评价与互动闭环**：用户确认收货后可发表图文评价，评价展示在商品详情页供其他用户参考；用户与商家在社区自由交流，持续沉淀社交关系。
 
 ## 四. 详细功能需求分解
 
 ### 模块一：商品中心模块
 
 承载平台所有商品的基础数据管理及店铺展示。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 商品管理 | 商品发布与编辑 | 支持富文本详情、多图轮播、多规格 SKU 设置（如颜色、尺码联动价格与库存）。 |
| | 商品状态流转 | 在售、下架、售罄。商品由审核通过的商家账号发布后默认可售，平台保留监管下架和一键上下架能力。 |
 | 店铺与评价 | 商家主页（店铺页） | 用户在商品页可进入商家主页，集中查看该商家的所有在售商品、店铺公告及在社区发布的广告动态。 |
| | 商品评价 | 商品详情页专属区域展示用户购买后的真实评价（文字/图片）。评价发布后默认展示，平台保留隐藏和治理能力。 |
 | 分类与品牌 | 树状分类 | 支持最多三级分类（如：食品 -> 零食 -> 坚果）。 |
 | | 营销标签 | 支持为商品添加"热卖"、"新品"、"特价"等视觉标签。 |
 | 商品搜索 | 综合检索 | 支持按名称关键词、分类、价格区间过滤。 |
 | | 列表排序 | 按综合、销量、价格升降序排列。 |
 | 个性化推荐 | 橱窗推荐 | 平台端手动配置首页"爆款"与"Banner轮播"。 |
 | | 猜你喜欢 | 在首页与商品详情底部，基于同类目随机或按销量倒序推荐关联商品（弱化算法复杂度，降低实训负担）。 |
 | 在线客服 | 客服入口 | 商品页设有客服入口，基于 WebSocket 实现用户与商家的即时聊天对答（支持文字、图片）。 |
 
 ### 模块二：交易与订单模块
 
 支撑从加购到售后的完整交易链路。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 购物车 | 基础操作 | 增删改查、批量选中并计算总价（前端实时联动，后端核验）。 |
 | | 状态同步 | 下架、库存不足的商品在购物车中自动标记为"失效"。 |
 | 订单生成 | 结算流程 | 校验库存 -> 选择收货地址 -> 使用优惠券/积分 -> 调起支付（接入沙盒微信/支付宝）。 |
 | | 履约方式选择 | 快递配送（填写收货地址）。 |
 | 订单流转 | 状态机管理 | 待支付 -> 待发货 -> 配送中 -> 待收货 -> 已完成 / 已取消 / 售后中。 |
 | | 订单评价 | 用户确认收货后，订单进入可评价状态，用户可发布图文评价。 |
 | | 自动化处理 | Celery 定时任务清理超时未支付订单；发货后 7 天自动确认收货。 |
 | 退货功能 | 退货流程 | 用户申请退货 -> 商家审核 -> 用户寄回商品（手动更新快递状态）-> 商家收货确认 -> 系统自动退款。退货全程不接入真实快递物流，商家与用户手动更新状态即可。 |
 
 ### 模块三：促销引擎模块（核心逻辑）
 
 设计灵活的规则引擎，提升平台转化率。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 限时特价 | 活动管理 | 管理端配置限时特价活动时间段及活动价格；前端展示倒计时。采用通用促销机制，使用商品常规库存，无需独立库存与限购逻辑。 |
 | 优惠券 | 券生命周期管理 | 类型包括无门槛券、满减券、新人自动发券。覆盖领取、使用、过期自动作废全流程，结算时由用户自行选择使用。 |
 | 满减满折 | 全局/分类活动 | 如"全场满 199 减 50"、"零食区满 3 件打 8 折"。 |
 | 独立拼团 | 社交拼单 | 拼团商品独立于普通商品体系。用户开团 -> 邀请好友参团 -> 达到人数即发货，超时未满自动退款。（拼团商品不可加入购物车，直接发起结算）。 |
 
 ### 模块四：社交与内容交互模块（项目特色）
 
 以"种草分享"为核心理念，构建低门槛、重体验的社区生态。社区支持图文内容，暂不支持视频。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 交流社区 | 多角色自由互动 | 社区是一个开放广场，消费者、商家均可自由发布内容并互动。未购买用户可发帖询问商品情况与使用体验；已购买用户可发布种草帖分享真实体验。支持点赞、评论。 |
 | | 帖子类型 | 帖子分为三类：普通交流贴（提问、讨论、分享）、种草贴（购买后关联商品分享体验）、商家广告贴。所有类型均可挂载商品链接。种草贴要求发帖人已购买该商品，系统自动校验订单后允许挂载。 |
 | | 帖子标签 | 支持为帖子添加话题标签（如 #零食测评 #穿搭分享），点击标签可查看同类帖子，帮助用户发现感兴趣的内容。 |
 | | 社区排序 | 社区帖子默认按"最新"或"热度（点赞数 + 评论数）"排序，无需复杂推荐算法。 |
 | | 个人主页 | 点击用户头像或昵称可进入其个人主页，查看该用户的基本资料及发帖记录 |
| | 互动合规管理 | 帖子、商品评价、帖子评论采用"先发布、后治理"机制，发布后默认公开显示。平台拥有隐藏、下架和账号治理权限；只有商家入驻申请需要事前审核。 |

## 补充：后续全量开发要求

当前项目已经具备电商平台基本功能闭环，后续开发不再以"只搭最小闭环"为目标，而应围绕需求逐项完全实现。

1. **前后端同步开发**：后端新增模型、字段、接口或状态后，必须同轮补充用户端或管理端调用入口、页面展示、错误提示和接口文档；不得长期只做后端接口而没有可验收前端。
2. **完整实现优先**：后续每个模块进入开发时，应按本需求书和实现设计书一次性补齐关键细节，包括状态流转、权限边界、异常场景、测试用例和页面交互。
3. **必要机制可补充**：若需求书未显式列出但完整功能必需，例如登出入口、数据刷新、分类树展示、用户个人中心、地址省市区与默认地址管理、操作日志、幂等和权限校验，应纳入实现。
4. **文档同步**：任何接口、状态、字段、权限或页面入口变化，都必须同步更新 `docs/api/*`、`docs/dev-setup.md`、`AGENTS.md` 和必要的设计文档。
 
 ### 模块五：会员与账户体系
 
 通过权益留住高价值用户。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 会员等级 | 等级体系 | 普通（注册即可）、银卡（累计消费满 500）、金卡（累计消费满 2000）、钻石（累计消费满 5000）。 |
 | | 等级权益 | 高等级会员享有免邮特权、积分加速（如 1.5 倍积分）、专属生日礼券等权益。 |
 | 积分账户 | 积分获取 | 每日签到（连续签到递增）、购物返利、种草积分奖励（他人通过你的帖子购买商品后获取）。 |
 | | 积分消耗 | 下单时按比例抵扣现金（如 100 积分 = 1 元）。 |
 | 个人信息 | 基本资料 | 用户可设置昵称、性别、头像等个人资料。 |
 | | 收货地址 | 支持管理多条收货地址（姓名、手机号、详细地址），下单时选择使用。 |
 
 ### 模块六：数据与后台管理模块
 
 为运营提供决策支撑。
 
 | 功能子模块 | 功能点 | 说明 |
 |-----------|-------|------|
 | 经营看板 | 核心数据指标 | 顶部展示今日大盘 GMV、新增订单数、新增用户数等关键指标。 |
 | | 趋势与排行 | 采用 Echarts 渲染：近7天销售额折线图、商品销量 TOP10 柱状图。 |
 | 权限管理 | RBAC 控制 | 平台运营、商家运营、普通用户三个角色，动态分配侧边栏菜单与按钮权限。 |
 | 数据导出 | 报表下载 | 支持将订单列表、用户列表导出为 Excel 文件（供对账使用）。 |
 
 > **编码提示**：后台数据分析视图应当覆盖多种可视化类型（如趋势图、对比图、分布图、排行图、占比图等），保障后续运营能从多维度洞察平台状况。具体视图清单以编码阶段设计为准，此处不逐一列举。
 
 **设计提示**：后续设计与实现过程中需充分考虑系统的可扩展性——模块划分、接口定义、数据模型等方面均应为未来功能迭代预留空间，降低扩展改造成本。
 
 ## 五. 非功能性需求建议（实训加分项）
 
 1. **高并发支持**：针对限时特价等大流量场景，说明使用 Redis 缓存预热与 Lua 脚本防止库存超卖。
 2. **安全性设计**：用户密码加盐哈希存储（Bcrypt）；JWT 进行身份鉴权；防范 SQL 注入与 XSS 攻击。
 3. **代码规范**：前端配置 ESLint + Prettier；后端基于 FastAPI 提供完整的 Pydantic 模型校验与 OpenAPI 文档。
 4. **事务一致性**：在订单生成与库存扣减环节，明确使用 MySQL 事务保障数据强一致性。
